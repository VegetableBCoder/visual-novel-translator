# Requirement-Driving - 文档驱动开发

核心原则：**基于需求文档进行开发，确保变更不会破坏现有功能。**

用户自己编写需求文档。你的职责是：阅读文档、自动在跟踪器中注册、通过与用户讨论完善不完整的需求、分析可行性、执行开发以及管理状态——全部自动完成。

## 用户输入

$ARGUMENTS

## 约定

- **需求目录：** `docs/requirements/`
- **跟踪器文件：** `docs/requirements/TRACKER.md`
- **编号：** `REQ-XXX` 格式，自动递增
- **状态流转：** `Todo` → `In Progress` → `Done`

### TRACKER.md 格式

```markdown
# 需求跟踪器

最新ID: N

| ID | 文档 | 状态 | 完成日期 | 备注 |
|----|----------|--------|-----------|-------|
| REQ-001 | xxx.md | Done | 2026-02-26 | - |
| REQ-002 | yyy.md | In Progress | - | - |
```

如果 TRACKER.md 不存在，则自动创建并初始化。在进行任何操作前始终读取最新状态。

## 模式

根据参数自动确定模式：

---

### 模式 1：查看状态

**触发条件：** 参数为 `status`、空，或用户要求查看状态。

1. 读取 TRACKER.md
2. 显示所有需求及其当前状态，按状态分组并统计数量
3. 高亮显示所有 `In Progress` 的需求

---

### 模式 2：接收需求文档

**触发条件：** 参数指向一个需求文档（文件名或路径）。

1. 读取指定的需求文档
2. 检查文档是否已在 TRACKER.md 中跟踪
    - 未跟踪 → 自动分配下一个 ID，以 `Todo` 状态添加到 TRACKER.md
    - 已跟踪 → 读取当前状态
3. **完善需求：** 如果文档简略或不完整，主动通过与用户讨论帮助用户充实内容：
    - 澄清模糊的功能描述
    - 如果缺少验收标准，建议补充
    - 识别未覆盖的边界情况和错误场景
    - 指出缺失的细节（API 定义、数据格式、性能要求等）
    - 评估对现有功能的影响
    - 评估技术可行性
4. 讨论结束后，帮助用户根据达成的共识更新文档
5. 询问用户：是否立即开始执行？

---

### 模式 3：执行需求

**触发条件：** 用户明确要求执行/开发某个需求，或在模式 2 分析后确认执行。

#### 预检查

- `In Progress` → 确认是否继续
- `Done` → 确认是否重做

#### 影响分析

- 审查 `docs/requirements/` 中所有 `Done` 和 `In Progress` 的需求文档
- 评估此次开发是否可能影响现有功能
- **如果有影响，明确告知用户哪些需求可能受到影响以及如何影响**

#### 设计实现计划

1. 根据需求文档和当前代码库，设计实现计划
2. 包括：方法概述、需修改/创建的文件、分步计划
3. 呈现给用户，等待确认

#### 执行开发

1. 自动将 TRACKER.md 状态更新为 `In Progress`
2. 逐步实施，在关键里程碑简要报告进度

#### 收尾

1. 自动将 TRACKER.md 状态更新为 `Done`，记录完成日期
2. 完成总结：修改的文件、验收标准检查清单、需要手动验证的项目

---

## 备注

- 所有 TRACKER.md 更新（添加记录、状态变更、ID 递增）都自动处理
- 执行需求时必须进行影响分析——不可跳过
- 保持文件系统与 TRACKER.md 引用之间的文档路径一致性
